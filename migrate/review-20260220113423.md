# Code Review: Legacy Banking CIF Application (src/)

## 🛡️ Security Audit

*   [ ] **[Critical]** Missing Concurrency Control in Transactions (Race Condition / Lost Update)
    *   *Location:* `src/main/java/com/banking/cif/service/BankingService.java:76` (`processTransaction`)
    *   *Why:* The `AccountDAO.findById(conn, transaction.getAccountId())` call does not use a pessimistic lock (e.g., `SELECT ... FOR UPDATE`). If two concurrent transactions read the balance before either commits, one of the updates will be overwritten, resulting in a lost update and incorrect account balances.
    *   *Recommendation:* Update `AccountDAO` to use `SELECT ... FOR UPDATE` when fetching accounts for transactions to ensure row-level locking, or implement optimistic locking with a version column.

*   [ ] **[Critical]** Missing Positive Amount Validation for Transactions
    *   *Location:* `src/main/java/com/banking/cif/service/BankingService.java:85` (`processTransaction`)
    *   *Why:* The transaction amount is not validated to be strictly positive. An attacker could submit a "DEPOSIT" of `-1000`, effectively withdrawing funds without triggering the insufficient funds check. Similarly, a "WITHDRAWAL" of `-1000` would add funds.
    *   *Recommendation:* Add strict validation `if (transaction.getAmount().compareTo(BigDecimal.ZERO) <= 0) throw new Exception("Amount must be positive");` at the start of `processTransaction`.

*   [ ] **[High]** Insecure Direct Object Reference (IDOR) / Missing Authentication
    *   *Location:* All Controllers (`AccountsController.java`, `CustomersController.java`, `TransactionsController.java`)
    *   *Why:* There are no authentication or authorization checks. Any user can manipulate the `id` parameter in API requests (e.g., `GET /api/v1/accounts/1`, `PUT /api/v1/accounts/2`) to view or modify other customers' data and accounts.
    *   *Recommendation:* Implement a robust session/token-based authentication mechanism and ensure that the current user owns the resource before allowing read or write operations.

*   [ ] **[High]** Stored Cross-Site Scripting (XSS) in Frontend Templates
    *   *Location:* `src/main/webapp/index.html` (e.g., lines 85, 102)
    *   *Why:* Underscore.js templates use `<%= value %>` for interpolation, which renders unescaped HTML. If a customer's name or email contains malicious JavaScript, it will execute when viewed by an employee.
    *   *Recommendation:* Change all `<%= ... %>` template interpolations to `<%- ... %>` to enforce HTML escaping.

*   [ ] **[Medium]** Predictable Identifiers
    *   *Location:* `AccountDAO.java:15` and `CustomersController.java:82`
    *   *Why:* Account numbers use `java.util.Random`, and CIF numbers use `System.currentTimeMillis()`. These are predictable and highly prone to collisions or enumeration attacks.
    *   *Recommendation:* Use a `SecureRandom` instance for generating account numbers, and consider using UUIDs or a database sequence for CIF numbers.

## 🐛 Logic & Correctness

*   [ ] **[Medium]** Customer Lookup by Name Fallback is Flawed
    *   *Location:* `src/main/java/com/banking/cif/action/CustomersController.java:62` (`show`)
    *   *Why:* If the `id` is not an integer, the system falls back to searching by name and returns the *first* customer in the results list (`model = customers.get(0);`). This means fetching `/api/v1/customers/john` will return an arbitrary "John", which is incorrect and leaks other users' data.
    *   *Recommendation:* Separate the search endpoint from the ID lookup endpoint. If searching by name, return a list, not a single arbitrary model.

## ♻️ Maintainability & Style

*   [ ] **[Low]** Controllers are tightly coupled with DAOs
    *   *Location:* `AccountsController.java:70`, `CustomersController.java:35`
    *   *Why:* The controllers perform direct data access and business logic validations (e.g., checking if an email exists, validating product codes).
    *   *Recommendation:* Move all business logic and DAO interactions into `BankingService.java`. The controllers should only handle HTTP routing and parameter parsing.

*   [ ] **[Low]** Hardcoded Database Credentials
    *   *Location:* `src/main/java/com/banking/cif/util/DBConnection.java:8`
    *   *Why:* The database URL, user, and password are hardcoded into the source code, preventing deployment to varying environments (e.g., PostgreSQL in production).
    *   *Recommendation:* Use an external properties file or environment variables to load the database configuration dynamically.

## 💡 Commendations

*   The RESTful URL design pattern is consistent.
*   Good use of `PreparedStatement` across all DAOs which prevents SQL injection attacks against database queries.
